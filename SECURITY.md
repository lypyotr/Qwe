# Безопасность приложения

Защита строится слоями. Шифрование — лишь один из них и НЕ главный: от «прочитать/
удалить базу через публичный ключ» защищает **контроль доступа (RLS)**, а не шифрование.

## Слой 1 — RLS (контроль доступа). ⚠️ Самое важное

Запустите **`security_setup.sql`** в Supabase → SQL Editor. Он включает Row Level
Security и задаёт политики «каждый видит только свои строки». Без этого публичный
ключ из клиента позволяет читать и удалять данные всех пользователей.

Проверка после запуска (в SQL Editor):

```sql
select relname, relrowsecurity from pg_class
  where relname in ('periods','user_settings','app_config');
-- relrowsecurity должно быть true у всех трёх
```

## Слой 2 — Усиление авторизации (Dashboard → Authentication)

- **Leaked password protection** — включить (проверка пароля по базе утечек
  HaveIBeenPwned). *Authentication → Policies / Passwords.*
- **Минимальная длина пароля** — ≥ 10, желательно с требованием символов разных
  классов.
- **Подтверждение email** — обязательное для регистрации (*Confirm email*).
- **Rate limiting** — ограничить частоту попыток входа/регистрации/сброса
  (*Authentication → Rate Limits*), чтобы исключить перебор.
- **CAPTCHA** (hCaptcha или Cloudflare Turnstile) на формах входа/регистрации
  (*Authentication → Bot and Abuse Protection*). Требует виджет в клиенте — могу
  добавить.
- **MFA / 2FA (TOTP)** — самая сильная защита аккаунта. Поддерживается Supabase;
  требует UI в приложении (привязка приложения-аутентификатора и запрос кода при
  входе). Могу реализовать.

## Слой 3 — Защита мастер-ключей и от удаления

То, что реально может «удалить базу», — это НЕ публичный ключ (его ограничивает
RLS), а:

- **service_role ключ** — секретный, обходит RLS. Он НЕ должен попадать в клиент
  или репозиторий (проверено — утечки нет). Храните только на сервере/в секретах.
- **Доступ к Supabase Dashboard** — это «ключ от всего». Включите **2FA на самом
  аккаунте Supabase** (account settings). Без этого взлом почты владельца = полный
  доступ к базе.
- **Бэкапы / Point-in-Time Recovery** — на случай, если данные всё же удалят
  (ошибка, компрометация владельца). Включите автоматические бэкапы/PITR в
  настройках проекта. В приложении уже есть авто-копия и экспорт, но серверный
  бэкап Supabase — отдельный независимый слой.

## Слой 4 (опционально, тяжёлый) — Сквозное шифрование (E2EE)

Шифровать данные ключом, производным от пароля пользователя, ДО отправки в
Supabase — тогда даже администратор Supabase не прочитает содержимое. Это
максимальная конфиденциальность, но с серьёзными компромиссами:

- **Забыли пароль → данные не восстановить** (нет сервера, который мог бы их
  расшифровать). Сброс пароля = потеря всех данных.
- Ломается серверная логика: бэкап «всей базы» владельцем, любая аналитика на
  сервере, отправка отчётов через функцию.
- Сложнее мультиустройство (нужна синхронизация ключа).

Для приложения учёта зарплаты обычно достаточно RLS + шифрование Supabase at-rest.
E2EE имеет смысл только если вы не доверяете самому Supabase. Делается, но взвесьте
риск потери данных.

### Включение E2EE (реализовано, под флагом, по умолчанию ВЫКЛ)

Схема: случайный ключ данных (DEK) шифрует содержимое; DEK завёрнут паролем входа
И recovery-кодом (envelope), поэтому смена пароля не теряет данные, а забытый
пароль восстанавливается кодом. В БД — только шифротекст; метаданные открыты.

Порядок (⚠️ сначала на ТЕСТОВОМ аккаунте, не на реальных данных):

1. Выполните `e2ee_setup.sql` в Supabase SQL Editor (добавляет колонки `cipher`
   и таблицу `user_keys` с RLS).
2. Зайдите тестовым аккаунтом, заполните пару периодов.
3. Профиль → «🔐 Шифрование данных» → «Включить шифрование». Подтвердите пароль,
   **сохраните recovery-код**.
4. Проверьте: перезагрузка (данные читаются), вход на втором устройстве (просит
   пароль → расшифровывает), смена пароля и вход по recovery-коду.
5. Только убедившись, что всё работает и есть свежий экспорт-бэкап — включайте на
   рабочем аккаунте.

Поведение: на новом устройстве после входа ключ разблокируется паролем входа и
кэшируется локально (чтобы перезагрузки не спрашивали пароль снова). Если ключ не
удалось получить — приложение показывает «🔒 Заблокировано» и не синхронизирует,
пока не разблокируете (пароль или recovery-код). Owner-бэкап «всей базы» при E2EE
содержит шифротекст (нечитаемый), это ожидаемо.
